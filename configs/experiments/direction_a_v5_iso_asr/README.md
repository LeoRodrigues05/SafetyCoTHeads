# Direction A v5 — iso-ASR sweep across reasoning-trace models

Successor to `19-v4-jbb-qwen3`. Three motivating problems and how v5 fixes
each:

| Problem in v4 | v5 fix |
|---|---|
| Llama-3.1-8B emits implicit-CoT prose; pathway/monitorability signals were noisy because the gibberish gate had to throw out 70% of `ships_top10` rows. | New policy models with **explicit `<think>` traces**: Qwen3-8B (thinking-mode), Qwen3-4B-Thinking, R1-Distill-Llama-8B. Llama-3.1 kept as a control. OLMo-2-7B SFT vs Instruct adds a *pre/post safety-training* axis. |
| Judge spuriously labels word-salad as `harmful_response` (33% raw for ships_top10 but 0% on clean rows). | New `gate_safety_by_coherence: true` config flag in `run_v4_jbb_judge.py`. Gibberish rows are skipped by the judge and auto-labeled `harmful_response=False`. Headline ASR is now `harmful_among_clean` everywhere. |
| SHIPS-top10 destroys coherence (`clean_rate=30%`); top10 is too aggressive. Need an iso-ASR view, not absolute. | Dose sweep per family: SHIPS `{3, 5, 8}`, neurons `{256, 512, 1024}`, steering `α ∈ {0.5, 1.0, 1.5}`. **Steering at α=1.0** is the iso-ASR anchor; the report's iso-ASR table reports `|ΔASR_clean|` vs the anchor and flags within-5pp matches. |

## Layout

```
configs/experiments/direction_a_v5_iso_asr/

  README.md
  <model_key>/                             <-- auto-generated by make_v5_configs.py
    01-ships-discovery.yaml                (skipped if reuse_discovery_*)

    gen/baseline.yaml
    gen/ships_top{3,5,8}.yaml
    gen/neurons_top{256,512,1024}.yaml
    gen/steering_a{0.5,1.0,1.5}.yaml

```

Model keys: `qwen3_8b`, `qwen3_4b_thinking`, `r1_distill_8b`,
`olmo2_7b_instruct`, `olmo2_7b_sft`, `llama31_8b_control`.

`olmo2_7b_sft` reuses `olmo2_7b_instruct`'s discovery artifacts (cross-checkpoint
test: do safety heads/neurons of the Instruct model still mediate refusal

`llama31_8b_control` reuses the existing v4 discovery files.

## Regenerating configs

```bash
python -m scripts.make_v5_configs \
    --matrix configs/experiments/direction_a_v5_iso_asr/matrix.yaml
# optional subset:
python -m scripts.make_v5_configs --matrix .../matrix.yaml --models qwen3_8b r1_distill_8b
```

Idempotent. Edit only `matrix.yaml`.

## Running

### Local (smoke test, one model, one cell)

```bash
python -m scripts.run_generation \
  --config configs/experiments/direction_a_v5_iso_asr/qwen3_8b/gen/baseline.yaml \
  --overrides dataset.n=8 batch_size=2
```

### Cluster (split per-model pipeline, preferred)

The original `direction_a_v5_per_model.sbatch` runs generation and all judging
serially inside one 24h allocation. The preferred path is now split:

1. `direction_a_v5_split_driver.sbatch` runs discovery + generation for one
   `MODEL_KEY`.
2. It writes a query+metric manifest under
   `runs/direction_a_v5/<model_key>/judge/_manifests/query_metric_tasks.jsonl`.
3. It submits `direction_a_v5_judge_query_metric.sbatch` as an array. Each
   manifest row is one query and one metric; workers may process multiple rows
   only to amortize judge-model load. Shards are written under
   `_query_metric_shards/`.
4. It submits `direction_a_v5_merge_report.sbatch` after the judge array. This
   merges shards back into the standard `judge_safety__*.jsonl`,
   `judge_pathway__*.jsonl`, `summary.json`, and `v5_report.md` outputs.

```bash
# Conservative default: 16 query+metric rows per worker, max 16 concurrent GPUs.
bash scripts/submit_direction_a_v5_split.sh \
  qwen3_8b qwen3_4b_thinking r1_distill_8b olmo2_7b_instruct

# Dependent model; if submitted in the same command after olmo2_7b_instruct,
# the helper adds an afterok dependency on the OLMo-Instruct split driver.
bash scripts/submit_direction_a_v5_split.sh olmo2_7b_instruct olmo2_7b_sft

# Tune parallelism/load amortization:
TASKS_PER_WORKER=32 MAX_ARRAY_CONCURRENCY=8 \
  bash scripts/submit_direction_a_v5_split.sh qwen3_8b
```

`TASKS_PER_WORKER` controls how many query+metric rows a single GPU worker
handles after loading the judge once. `MAX_ARRAY_CONCURRENCY` caps simultaneous
judge workers per model.

### Cluster (legacy full per-model pipeline)

One sbatch per model. OLMo-SFT and Llama-control must run **after** their
discovery parent has finished.

```bash
# Independent first wave:
sbatch --export=ALL,MODEL_KEY=qwen3_8b           scripts/sbatch/direction_a_v5_per_model.sbatch
sbatch --export=ALL,MODEL_KEY=qwen3_4b_thinking  scripts/sbatch/direction_a_v5_per_model.sbatch
sbatch --export=ALL,MODEL_KEY=r1_distill_8b      scripts/sbatch/direction_a_v5_per_model.sbatch
sbatch --export=ALL,MODEL_KEY=olmo2_7b_instruct  scripts/sbatch/direction_a_v5_per_model.sbatch

# Dependent (need discovery from above):
sbatch --export=ALL,MODEL_KEY=olmo2_7b_sft       scripts/sbatch/direction_a_v5_per_model.sbatch
sbatch --export=ALL,MODEL_KEY=llama31_8b_control scripts/sbatch/direction_a_v5_per_model.sbatch  # reuses v4 artifacts already on disk
```

Each job: discovery (≤3 steps) → 10 generation cells → v4 judge (single load,
all conditions) → v5 markdown report under
`runs/direction_a_v5/<model_key>/judge/v5_jbb_report.md`.

## What the report contains

1. **Headline** — `clean_rate`, `ASR_clean` (gibberish-gated), raw harmful, refusal.
2. **Iso-ASR comparison** — `|ΔASR_clean|` vs `steering_a1.0` anchor, with ✓ flag for within-5pp matches.
3. Coherence gate (gibberish histogram + repeat3 / compression-ratio diagnostics).
4. Basic 5-metric safety scoreboard (legacy, gate-aware).
5. Pathway vector (8 means).
6. Dominant pathway distribution.
7. CoT-monitorability gap (ASR_final − ASR_cot_pred).

## Notes on individual models

- **Qwen3-8B**: `chat_overrides.enable_thinking: true` activates the `<think>` block in the chat template. `max_new_tokens=1024` to give the trace room. Plumbed via `render_chat(..., chat_template_kwargs=...)`.
- **Qwen3-4B-Thinking-2507**: thinking-only variant; no toggle needed.
- **R1-Distill-Llama-8B**: already configured; `segmentation.py` handles `<think>` tags natively.
- **OLMo-2 (both)**: prose-CoT, no `<think>` tags. Pathway segmentation falls back to sentence splitting on the response. Primary value is the **training-stage axis**.
- **Llama-3.1-8B-Instruct (control)**: re-judged with `gate_safety_by_coherence=true` so the ASR_clean number is apples-to-apples with the new models. Uses existing v4 generation outputs by symlink/path.

## Code changes vs v4

- `src/safety_cot_heads/generation/{prompts,generate}.py` — `chat_template_kwargs` plumbing for Qwen3 `enable_thinking`.
- `scripts/run_v4_jbb_judge.py` — `gate_safety_by_coherence` flag: skip gibberish rows in safety judge, synthesize `judge_flat` with `label_present=False, skipped_gibberish=True`.
- `scripts/run_generation.py` — forwards `chat_overrides` from cfg.
- `scripts/make_v4_jbb_report.py` — new section 0 "Headline" + section 1 "Iso-ASR comparison"; `--iso-anchor` + `--title` CLI flags.
- `scripts/make_v5_configs.py` — new matrix expander.
- `scripts/sbatch/direction_a_v5_per_model.sbatch` — new per-model driver.

## Adding a model

1. Add an entry to `configs/models.yaml`.
2. Append to `matrix.yaml > models` with `model_ref:` + `max_new_tokens` + `batch_size` (+ optional `chat_overrides` / `reuse_discovery_*`).
3. `python -m scripts.make_v5_configs --matrix .../matrix.yaml`
4. `sbatch --export=ALL,MODEL_KEY=<key> scripts/sbatch/direction_a_v5_per_model.sbatch`
